# Sentry Sound Academy SLB Structure V1

## Purpose

This document defines the Sentry Sound Academy SLB structure.

These are NOT school curriculum SLBs.

These are academy competency SLBs designed for:
- music industry training
- platform-aligned operations
- outcomes-based learning
- competency verification
- practical execution
- evidence capture
- remediation
- future runtime execution

---

# Definition

An Academy SLB is a small, assessable, evidence-capable competency block.

It represents one clear unit of learning or execution that can be:
- taught
- practised
- assessed
- remediated
- verified
- tracked
- reused
- executed by the runtime later

---

# Core Rule

Every SLB must connect to:

1. A module
2. A competency level
3. One or more outcomes
4. Evidence requirements
5. Assessment logic
6. Remediation logic
7. Industry relevance
8. Platform/backend relevance where applicable

---

# SLB Types

## 1. Concept SLB
Learner understands a key idea, term, principle, standard, role, body, or process.

Example:
- Understand what ISRC means
- Understand what SAMRO does
- Understand what master rights are

## 2. Workflow SLB
Learner can follow or execute a defined operational process.

Example:
- Prepare a registration workflow
- Complete a metadata capture process
- Follow a release approval workflow

## 3. Evidence SLB
Learner can create, collect, validate, or submit evidence.

Example:
- Prepare a split sheet
- Compile a session log
- Prepare a registration evidence pack

## 4. Platform SLB
Learner can perform a platform-aligned task inside or related to Sentry Sound Platform logic.

Example:
- Capture contributor metadata
- Track compliance status
- Review royalty workflow state

## 5. Assessment SLB
Learner demonstrates competence through a task, quiz, case study, simulation, or practical submission.

Example:
- Validate a metadata pack
- Identify missing registration fields
- Resolve a rights conflict scenario

## 6. Remediation SLB
Learner corrects a failed or weak competency area.

Example:
- Correct incorrect contributor split logic
- Rebuild incomplete metadata
- Reattempt failed compliance workflow

## 7. Governance SLB
Learner understands or executes control, audit, approval, review, or oversight logic.

Example:
- Review an audit trail
- Apply workflow approval rules
- Identify compliance risk states

# Required SLB Fields

Each SLB must define:

- SLB ID
- SLB Name
- Module ID
- Module Name
- Academy Layer
- Competency Level
- SLB Type
- Version
- Status
- Estimated Duration
- Prerequisites
- Target Learner Profile

---

# Outcome Fields

Each SLB must define:

- Primary outcome
- Supporting outcomes
- Observable capability
- Success criteria
- Failure indicators

---

# Evidence Fields

Each SLB must define:

- Required evidence
- Evidence format
- Evidence quality rules
- Evidence verification method
- Evidence storage expectation

---

# Assessment Fields

Each SLB must define:

- Assessment method
- Assessment task
- Pass criteria
- Remediation trigger
- Reassessment method

---

# Platform Alignment Fields

Where applicable, each SLB must define:

- Related Sentry Sound Platform entity
- Related workflow
- Related metadata fields
- Related compliance status
- Related backend logic
- Related operational risk

If no platform relationship exists, mark as:
Not applicable.

---

# Standards Alignment Fields

Each SLB should include placeholders for:

- SAQA relevance
- SETA relevance
- NQF level consideration
- Notional hours consideration
- Credit relevance consideration
- International standard relevance
- Industry body relevance

Formal mapping happens later.

---

# Runtime Compatibility Fields

Each SLB should later support:

- learner event tracking
- competency state updates
- remediation routing
- evidence submission events
- assessment scoring
- verification status
- reporting outputs

---

# SLB Design Principles

Academy SLBs must be:

- small enough to assess
- clear enough to teach
- practical enough to execute
- measurable enough to verify
- structured enough for future runtime execution
- aligned enough for future certification
- reusable across modules where appropriate
