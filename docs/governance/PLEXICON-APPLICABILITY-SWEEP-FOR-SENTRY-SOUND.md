## 4. Product UX Implications

Sentry Sound UX should stay product-first. The user should feel they are managing music-rights work, not operating Plexicon doctrine.

Guidance:

- Present calm operational surfaces that help users understand what exists, what is missing, what can be fixed now, and what is only visible for review.
- Use natural workflow language instead of raw doctrine terms.
- Keep backend intelligence supportive: it should shape summaries, gaps, warnings, and next actions, but not dominate the page.
- Keep frontend interpretation distinct from backend truth ownership.
- Avoid doctrine-heavy UX labels such as OEF stages, semantic orchestration, governance chains, primitive registers, or interpretation models.
- Keep progress/completeness visuals clearly contextual and non-certifying.
- Preserve cognitive-load boundaries: surface detail when it changes user action, risk, review, authority, or uncertainty.

## 5. Backend / Runtime Implications

Backend and runtime work should remain source-of-truth oriented and bounded.

Guidance:

- Preserve source-of-truth ownership in Sentry Sound modules: works, contributors, splits, supporting materials, completeness, finance, actions, and future evidence/submission logic.
- Preserve lineage and evidence basis when deriving context.
- Retrieve existing work, asset, contributor, completeness, material, and action context before generating new guidance.
- Keep governance before autonomous advancement: no hidden progression, no autopilot, no AI approval, no submission-like claims without the proper chain.
- Treat telemetry and activity as evidence requiring interpretation, not truth.
- Build bounded operational progression only: small runtime slices with explicit allowed data, prohibited behavior, validation method, and documentation update.
