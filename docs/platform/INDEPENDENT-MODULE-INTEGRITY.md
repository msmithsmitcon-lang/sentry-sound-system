# Independent Module Integrity

## Purpose

This document locks the Sentry Sound platform principle that every paid or available module must deliver clear standalone operational value within its promised scope.

Sentry Sound may encourage users to expand into other modules, tiers, services, automations, or intelligence layers, but that expansion must be based on enhancement rather than artificial dependency.

## Core Principle

Independent Module Integrity:

Each paid module must deliver clear standalone operational value within its promised scope. Cross-module integration may enhance, automate, orchestrate, deepen visibility, or improve intelligence, but must not intentionally cripple core workflows, hide essential records, or force users into unrelated modules to complete the basic job they paid for.

Supporting rule:

Upgrade pressure must be additive, not punitive.

## What This Means

A module can depend on platform foundations such as authentication, workspace ownership, permissions, audit, and canonical records. It must not depend on unrelated commercial modules for its core promise.

Examples:

- Works/Songs must remain useful without finance, AIL, submission, royalty, or release modules.
- Finance must remain useful without royalty ingestion, release planning, artist intelligence, or submission tooling.
- File Vault and Supporting Materials must remain useful as workspace-owned reference and document surfaces without claiming legal readiness or requiring a future evidence/legal module for basic reference value.
- Artist Intelligence Layer, when built, must remain advisory and enhancing. It must not be required for basic work capture, finance use, file references, or submission visibility.
- Release/Submission modules must be clear about their scope and must not imply regulator acceptance, legal clearance, or official authority unless the governed backend and external process support that claim.

## Module Boundary Template

Every module should define these boundaries before production entitlement or packaging decisions:

1. Core standalone promise
   - What operational job can the user complete with this module alone?

2. Optional integrations
   - Which other modules make the experience richer, faster, more automated, or more visible?

3. Explicit exclusions
   - What does this module not claim, certify, automate, or decide?

4. Upgrade-safe language
   - Which wording explains enhancement without implying hostage workflows?

5. Data access/export expectations
   - What records can the user view, manage, and reasonably export within the module's promised scope?

## Upsell And Upgrade Language

Use:

- Add cross-module visibility
- Automate this workflow
- Connect this to finance/submissions/releases
- Unlock deeper reporting
- Coordinate this across your workspace

Avoid:

- You cannot continue
- Your work is incomplete unless you upgrade
- Your data is locked
- Submission-ready when only references exist
- AI knows what to do without reviewable user control

## Data Access Principle

Basic access to user-owned records must not be treated as a hostage feature.

Higher tiers may add:

- scale
- storage
- team collaboration
- advanced exports
- governed automation
- deeper reporting
- integrations
- orchestration
- strategic intelligence
- enterprise support

Higher tiers must not remove the user's ability to understand, access, and operate the core records created under the module they paid for, except where required by legitimate security, billing, legal, or abuse-prevention rules.

## AI And AIL Boundary

AI, AIL, assistants, and future orchestration layers must remain advisory, reviewable, and enhancing.

They may:

- explain operational context
- suggest actions
- recommend links between modules
- identify gaps
- propose workflow items
- deepen strategic intelligence

They must not:

- silently overwrite user truth
- become required for basic module operation
- hide essential module records behind AI access
- create completed actions without review
- pressure upgrades through fear or opaque judgment

## Entitlement And Packaging Guidance

Entitlement gates should control access to governed capabilities, scale, automation, integrations, and mature backend features.

They should not be designed to create artificial friction inside a paid module's core workflow.

When a feature is unavailable, blocked, deferred, or plan-limited, the product must distinguish:

- plan limitation
- backend capability not implemented
- permission issue
- workspace setup issue
- evidence/readiness issue
- legal/rights authority issue

These states must not be collapsed into vague upgrade pressure.

## Source-Of-Truth Relationship

This principle should guide:

- subscription and entitlement design
- module packaging
- dashboard capability visibility
- route/API ownership planning
- user-facing upgrade prompts
- future AI/AIL/orchestration design
- service offering language
